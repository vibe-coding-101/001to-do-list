## 2. STRIDE 威胁分析

### 2.1 Spoofing（身份伪造）

| 威胁场景 | 影响范围 | 可能性 | 优先级 |
|----------|----------|--------|--------|
| 无身份认证机制，任意用户可访问 | 低（单用户本地应用，无需认证） | - | N/A |

**分析结论**：本应用为纯前端单用户本地应用，无用户身份认证需求，不存在身份伪造威胁。

### 2.2 Tampering（数据篡改）

| 威胁场景 | 影响范围 | 可能性 | 优先级 |
|----------|----------|--------|--------|
| 用户通过浏览器开发者工具直接修改 LocalStorage 数据 | 应用功能异常 | 中 | P1 |
| 恶意脚本通过 XSS 注入篡改 LocalStorage | 数据丢失/污染 | 低 | P0 |
| 第三方库 @tanstack/vue-virtual 被供应链攻击植入恶意代码 | 数据泄露/篡改 | 低 | P1 |

**详细分析**：

1. **LocalStorage 直接篡改**
   - 攻击向量：用户或恶意脚本通过 `localStorage.setItem()` 修改存储数据
   - 现有防护：`storage.ts` 中 `loadTasks()` 包含 `isValidTask()` 格式验证
   - 风险残留：验证仅检查字段类型，未验证值域（如 priority 可能被设为非法值）

2. **XSS 注入篡改**
   - 攻击向量：任务文本中注入恶意脚本
   - 现有防护：Vue 模板使用 `{{ task.text }}` 自动转义，无 XSS 风险
   - 风险确认：TaskItem.vue 第 53 行、TaskInput.vue 第 47 行均使用插值语法，Vue 自动转义

3. **供应链攻击**
   - 攻击向量：@tanstack/vue-virtual 包被植入恶意代码
   - 现有防护：无
   - 风险等级：低（TanStack 为知名开源组织，维护活跃）

### 2.3 Repudiation（抵赖）

| 威胁场景 | 影响范围 | 可能性 | 优先级 |
|----------|----------|--------|--------|
| 无审计日志，无法追溯数据操作历史 | 低（单用户本地应用） | - | N/A |

**分析结论**：本应用为单用户本地应用，无审计追溯需求，不存在抵赖威胁。

### 2.4 Information Disclosure（信息泄露）

| 威胁场景 | 影响范围 | 可能性 | 优先级 |
|----------|----------|--------|--------|
| LocalStorage 数据被同域其他脚本读取 | 数据泄露 | 中 | P2 |
| 浏览器历史记录/缓存泄露任务内容 | 数据泄露 | 低 | P2 |
| 控制台日志泄露敏感信息 | 数据泄露 | 低 | P2 |

**详细分析**：

1. **同域脚本读取**
   - 攻击向量：若部署环境存在其他应用或 XSS 漏洞，可读取 LocalStorage
   - 现有防护：无（依赖部署环境安全）
   - 建议：若未来扩展为多用户应用，需加密敏感数据

2. **浏览器历史/缓存**
   - 攻击向量：任务内容可能出现在 URL 参数（当前未使用）
   - 现有防护：应用未在 URL 中传递任务内容
   - 风险确认：无

3. **控制台日志**
   - 攻击向量：`console.error()` 可能泄露错误详情
   - 现有代码审查：
     - `task.ts` 第 57 行：`console.error('Failed to load tasks:', error)` - 仅打印错误对象
     - `storage.ts` 第 52 行：`console.error('Failed to parse tasks:', error)` - 同上
   - 风险等级：低（无敏感数据直接打印）

### 2.5 Denial of Service（拒绝服务）

| 威胁场景 | 影响范围 | 可能性 | 优先级 |
|----------|----------|--------|--------|
| LocalStorage 存储空间耗尽 | 应用崩溃 | 中 | P0 |
| 大量任务数据导致渲染阻塞 | 性能下降 | 中 | P0（v0.4.0 已通过虚拟滚动解决） |
| 恶意输入触发无限循环 | 浏览器卡死 | 低 | P1 |

**详细分析**：

1. **LocalStorage 配额耗尽**
   - 攻击向量：大量任务写入导致 `QuotaExceededError`
   - 现有防护：`storage.ts` 第 73-76 行已捕获 `QuotaExceededError` 并抛出友好错误
   - 风险残留：未实现清理旧任务逻辑（代码注释 TODO 标记）

2. **大数据量渲染阻塞**
   - 攻击向量：10000+ 任务全量渲染
   - 现有防护：v0.4.0 引入虚拟滚动，仅渲染可视区域
   - 风险缓解：虚拟滚动方案已解决此问题

3. **无限循环**
   - 攻击向量：computed 依赖循环、watch 无限触发
   - 现有代码审查：未发现循环依赖风险
   - 风险等级：低

### 2.6 Elevation of Privilege（权限提升）

| 威胁场景 | 影响范围 | 可能性 | 优先级 |
|----------|----------|--------|--------|
| 无权限模型，无权限提升风险 | N/A | - | N/A |

**分析结论**：本应用为单用户本地应用，无权限分级，不存在权限提升威胁。

## 3. 安全加固要求

### P0 — 必须实现

- [x] **[已满足] XSS 防护**：Vue 模板使用插值语法 `{{ }}` 自动转义，无 `v-html` 直接渲染用户输入
- [x] **[已满足] 输入验证**：`validation.ts` 实现任务文本长度验证（最大 200 字符）和空值检查
- [x] **[已满足] 数据格式验证**：`storage.ts` 的 `isValidTask()` 验证 LocalStorage 数据格式
- [x] **[已满足] 存储配额处理**：`storage.ts` 捕获 `QuotaExceededError` 并友好提示
- [ ] **[待增强] 值域验证**：`isValidTask()` 需增加 `status` 和 `priority` 枚举值验证

**值域验证修复示例**（`src/frontend/src/utils/storage.ts`）：

```typescript
const VALID_STATUSES = ['completed', 'uncompleted'] as const
const VALID_PRIORITIES = ['high', 'medium', 'low'] as const

function isValidTask(task: any): task is Task {
  return (
    typeof task === 'object' &&
    task !== null &&
    typeof task.id === 'string' &&
    typeof task.text === 'string' &&
    VALID_STATUSES.includes(task.status) &&
    VALID_PRIORITIES.includes(task.priority) &&
    typeof task.createdAt === 'number' &&
    typeof task.updatedAt === 'number' &&
    (task.completedAt === null ||
      task.completedAt === undefined ||
      typeof task.completedAt === 'number')
  )
}
```

### P1 — 建议实现

- [ ] **[待实现] npm 依赖完整性验证**：前端工程师在安装 @tanstack/vue-virtual 后应验证包完整性
  - 验证方式：`npm audit` 检查已知漏洞
  - 锁定版本：`package.json` 使用精确版本号 `3.13.23` 或 `^3.13.0`

- [ ] **[待实现] 存储空间管理**：当 LocalStorage 配额不足时，自动清理最旧的任务
  - 实现位置：`storage.ts` 第 75-76 行 TODO 注释处
  - 策略：删除 `createdAt` 最早且已完成的任务

- [ ] **[待确认] 生产环境移除 console**：建议在生产构建时移除 `console.error` 日志
  - 实现方式：Vite 配置 `build.minify` 选项或使用 `terser` 的 `drop_console` 选项

### P2 — 后续跟进

- [ ] **[低优先级] 敏感数据加密**：当前无敏感数据，若未来增加隐私相关任务内容，考虑使用 Web Crypto API 加密
- [ ] **[低优先级] Content Security Policy**：部署时配置 CSP 头，限制脚本来源

## 6. 总结

### 6.1 威胁建模结论

v0.4.0 版本为纯前端本地应用，整体安全风险较低。主要安全考量如下：

| 威胁类型 | 风险等级 | 主要发现 |
|----------|----------|----------|
| Spoofing | 无 | 单用户应用，无身份认证需求 |
| Tampering | 低 | XSS 已通过 Vue 插值语法防护；值域验证需增强 |
| Repudiation | 无 | 单用户应用，无审计需求 |
| Information Disclosure | 低 | 无敏感数据；控制台日志风险可控 |
| Denial of Service | 中 | 存储配额处理已实现；虚拟滚动解决大数据量问题 |
| Elevation of Privilege | 无 | 无权限模型 |

### 6.2 核心安全要求

1. **必须实现**：增强 `isValidTask()` 的枚举值验证
2. **建议实现**：npm audit 检查依赖安全、实现存储空间自动管理
3. **后续跟进**：生产环境日志处理、CSP 配置

### 6.3 上线建议

**可上线**。v0.4.0 版本满足基本安全要求，无高危漏洞。建议在上线前完成 P0 级值域验证增强，并在部署时运行 `npm audit` 确认依赖安全。
