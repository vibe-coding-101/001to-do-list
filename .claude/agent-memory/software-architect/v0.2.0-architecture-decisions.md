## v0.2.0 架构设计决策

### 优先级选择功能

**决策**: 数据模型已支持priority字段,只需新增UI组件

**理由**:
- v0.1.1版本数据模型已包含完整的priority字段定义
- Task类型已定义TaskPriority = 'high' | 'medium' | 'low'
- LocalStorage验证函数已包含priority检查
- 只需新增PrioritySelector.vue组件即可完成UI部分

**实现方案**:
- 创建独立的PrioritySelector.vue组件,使用分段控制器(Segmented Control)样式
- 在TaskInput.vue中集成,用于创建任务时选择优先级
- 在TaskItem.vue编辑模式中集成,用于修改优先级
- 默认优先级为'medium',在createTask中处理

### 搜索功能架构

**决策**: 新增独立的SearchStore + 在FilterStore中集成搜索逻辑

**理由**:
- 搜索和过滤是正交的两个维度,应该分离关注点
- SearchStore只管理搜索状态(searchQuery)
- FilterStore的filteredTasks getter负责组合过滤和搜索逻辑
- 这种设计使得每个Store职责单一,易于测试和维护

**为什么不在TaskStore中处理搜索**:
- TaskStore应该只关心任务数据的CRUD操作
- 搜索属于视图层逻辑,不属于数据模型层
- 分离后可以更灵活地扩展搜索功能(如将来支持高级搜索)

**为什么使用computed而非单独的searchTasks方法**:
- Vue 3的computed自动缓存结果,性能更好
- 响应式依赖自动追踪,searchQuery变化时自动重新计算
- 代码更简洁,无需手动调用更新方法

**技术要点**:
- 搜索使用200ms防抖,避免频繁计算
- 搜索不区分大小写,使用toLowerCase()
- 搜索和过滤是AND关系(同时满足两个条件)
- 搜索结果仍按创建时间倒序排列

### 数据兼容性策略

**决策**: 在loadTasks()中处理旧数据兼容

**理由**:
- LocalStorage中可能存在没有priority字段的旧数据
- 在数据加载时统一处理,避免到处写兼容代码
- 使用空值合并运算符(priority || 'medium')自动设置默认值

**为什么不需要显式数据迁移**:
- priority字段是可选的,旧数据没有此字段不影响核心功能
- TypeScript类型定义中priority为必填,但运行时数据可能缺失
- 使用渐进增强策略,新数据有priority,旧数据自动补全

### 性能优化策略

**决策**: 使用防抖 + computed缓存

**理由**:
- 搜索输入使用200ms防抖,减少filteredTasks重新计算次数
- computed自动缓存,只有依赖变化时才重新计算
- 时间复杂度O(n),对于1000个任务完全可以接受

**为什么暂不实现虚拟滚动**:
- MVP版本优先简单实现
- 1000个任务的DOM性能在现代浏览器中可接受
- 留作后续优化项(待确认项#1)

### 组件设计原则

**决策**: 优先级选择器作为独立组件

**理由**:
- 遵循单一职责原则,组件只做一件事
- 可复用(创建和编辑都用)
- 易于测试和维护
- 符合Vue 3组合式API的设计理念

**为什么不用下拉选择**:
- 分段控制器更直观,点击即切换
- 移动端触摸体验更好
- 视觉上更突出优先级的重要性(颜色区分)
